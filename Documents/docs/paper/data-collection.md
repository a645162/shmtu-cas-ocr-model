# 2. 数据采集

## 2.1 数据来源

本文的验证码数据来源于上海海事大学 CAS 统一认证平台的验证码接口。该接口在每次登录请求时动态生成一张算术验证码图片，图片格式为 JPEG，分辨率为固定尺寸，包含一个「数字-运算符-数字 = 结果」格式的算术表达式。

## 2.2 采集管线设计

由于验证码接口不提供标注信息，本文采用「先 OCR 后验证」的半自动标注策略，构建了如图 1 所示的自动化采集管线。

### 2.2.1 采集流程

采集管线包含以下步骤：

1. **验证码请求**：通过模拟 CAS 登录流程，向验证码接口发送 HTTP 请求获取验证码图片。
2. **OCR 预标注**：将获取的验证码图片发送至已有的 OCR 推理服务（V1 三模型方案），获取初步识别结果。
3. **反向验证**：将 OCR 识别结果提交至 CAS 登录接口，若登录成功则说明识别结果正确，可作为标注使用。
4. **标注存储**：将验证通过的图片与对应的表达式标注以 jpg+json 配对形式保存。

### 2.2.2 并发采集

为提高采集效率，系统采用多进程并发架构：

```bash
python -m cas_ocr_model.datasets.dataset_collector \
    --backend restful \
    --ocr-url http://127.0.0.1:21600 \
    --output ./dataset \
    --count 5000 \
    --processes 4 \
    --per-process 8
```

其中，`--processes 4 --per-process 8` 表示启动 4 个采集进程，每进程 8 个并发请求，总计 32 个并发采集通道。在典型网络环境下，每分钟可采集约 200–500 条有效标注数据。

### 2.2.3 断点续采

采集过程支持断点续采：系统记录已采集的文件编号，重启后自动跳过已存在的样本，避免重复采集。

## 2.3 数据集统计

### 2.3.1 数据规模

采集并标注的验证码数据集统计信息如下：

| 项目 | 数值 |
|---|---|
| 总样本数 | 约 5000+ |
| 训练集比例 | 90% |
| 验证集比例 | 10% |
| 图片格式 | JPEG, 灰度图 |
| 标注格式 | JSON (expression, answer, digit_left, operator, digit_right) |

### 2.3.2 类别分布

**数字类别 (digit_left, digit_right)**：各 10 类 (0–9)，分布基本均匀，因 CAS 系统随机生成。

**运算符类别 (operator)**：3 类 (`+`, `-`, `*`)。实际观察中，减法 `-` 出现频率略高于加法 `+` 和乘法 `*`，因此训练时采用类别权重 `[1.0, 1.05, 1.1]` 进行轻微平衡。

### 2.3.3 数据格式

每条样本由一对文件组成：

```
dataset/
  00000000.jpg        # 验证码原始图片
  00000000.json       # 标注信息
```

JSON 标注格式：

```json
{
  "expression": "9 - 2 = 7",
  "answer": "7",
  "digit_left": "9",
  "operator": "-",
  "digit_right": "2"
}
```

## 2.5 数据预处理

训练前对原始图像进行以下预处理：

1. **灰度化**：将 RGB 图像转换为单通道灰度图。
2. **二值化**：采用最小通道 Otsu 方法 (`min_channel_otsu`)，对 RGB 三通道分别计算 Otsu 阈值后取最小值，有效保留浅色字符。
3. **归一化**：将像素值归一化至 [0, 1] 区间。
4. **尺寸调整**：将图像缩放至 64×192 像素。

二值化模式对比：

| 模式 | 方法 | 适用场景 |
|---|---|---|
| `min_channel_otsu` | 三通道 Otsu 取最小阈值 | 默认推荐，保留浅色字符 |
| `gray_otsu` | 灰度 Otsu | 通用基线 |
| `adaptive` | 自适应阈值 | 不均匀光照 |
| `fixed` | 固定阈值 200 | 高对比度场景 |
